Introduction

This web application is a JavaScript based assembly simulator. In this section, you will find a mile high overview of assembly. For those already familiar with assembly, jump ahead to the Instruction Definitions section. For a more in-depth and complete understanding of assembly try the wikipedia entry or a variety of the many resources available online.

Assembly Overview

Assembly is a programming language that can be easily converted to a computer processor's language. Generally, the conversion consists of a direct one to one mapping from text form to a sequence of bytes. This sequence of bytes is generally known as machine code which the processor can read directly. Byte code on the other hand is a sequence of bytes that runs on a virtual processor. Given this application uses a virtual processor, the term used in the main memory table of the emulator page is termed byte code instead of machine code.

Generally, assembly is dynamically generated via a compiler that transforms some low level source code such as C to a given assembly language such as ARM or x86. Modern compilers do a really good job of generating optimized assembly code so there are few opportunities where it is truly necessary to write assembly.

Why Learn Assembly?

So why learn assembly? Learning assembly will provide you with an in depth knowledge of how programs operate within the constraints of hardware. To name a few specific reasons:

  • Assembly reveals the performance penalty of function calls and the potential risk of stack overflow.
  • The presence of condition registers that consequently determine branching and conditionals.
  • The challenges and necessity of memory management.
  • And more...

Hardware Overview

Assembly is intimately associated with the hardware it is running on. At a bare minimum, a majority of assembly languages typically interact with the following three hardware components:

  1. The central processing unit (CPU) is the brain behind the computer (referred to throughout this tutorial as simply the processor). In reality, the CPU recognizes a fixed number of instructions and rotely matches an instruction from the stream of byte coded instructions being fed to it with one of its recognized instructions and executes that instruction.

    For this simulator, the CPU is abstracted away in JavaScript and is only visible to you via its memory footprint (see two components below). For more information on CPU's reference the wikipedia entry.

    There are a wide variety of processors present hence there is an equally diverse set of assembly languages present. The assembly language used in this emulator is a 16-bit home brew assembly that resembles those used in basic micro controllers.

  2. Main memory is the set of memory in which the machine code is stored. Main memory generally has the following features/purposes:

    1. Stores the assembly program itself.
    2. Features a stack at the bottom. The stack is used for function calls and can be used to store temporary data. Stack overflow is when the stack gets too large and overflows into the program or stored data values.
    3. Can be used by the assembly program to store data while the program is being run.

    On your personal machine, this memory is your computers' RAM and the memory management techniques become much more complicated.

    This simulator has 512 16-bit memory slots (this may increase to 1024 or more upon greater demand).

  3. Registers are a small set of memory that reside in the processor. In this simulator there are four types of registers and seven registers total:

    1. The Program Counter (PC) which points to the location in memory where the next assembly instruction to execute resides. The SP is bound by the main memory size which in this case is 9 bits wide.
    2. The Stack Pointer (SP) which points to the location in memory where to deposit the next value in the stack. The SP is bound by the main memory size which in this case is 9 bits wide.
    3. The registers R0, R1, R2 and R3 are 16-bit general purpose registers that are used to store or output data.
    4. The ZCNO register is a four bit condition register which has conditionals that get set as a result of given instructions. The conditions are detailed in the table below:
      Flag Purpose
      Z Z stands for zero and gets asserted for given instructions that result in zero.
      C C stands for carry and gets asserted for given instructions where an arithmetic or logical carry get generated. Logical carry includes left shifting a value who's most significant bit is one and arithmetic carry includes adding two values which exceeds the given 16 bit limit.
      N N stands for negative and gets asserted for given instructions who's results' most significant bit is one. See 2's complement for explanation.
      O O stands for overflow and gets asserted for given instructions that result in arithmetic overflow. To summarize arithmetic overflow is when the result exceeds the 16-bit limit and the result gets truncated at 16 bits.

    The PC and SP are not directly accessible and can only be manipulated as the result of other instructions that require modifying the PC or SP.

Instruction Syntax

Every assembly language has its own, albeit similar, syntax. The syntax of an assembly language is the grammar the assembly program must be written in accordance to in order for the program to be legally transformed into machine code. The syntax is defined in the following two sections.

Syntax Tokens

This assembly programming language (as with all programming languages) is composed entirely from a combination of a set of tokens. Tokens are the building blocks of a programming language. The six different token types used here are detailed in the table below:

Green indicates that the token can only be used for arguments.
Pink indicates that the token is optional and does not impact run time.
Name Description
Instruction The instruction is one of the commands that the processor recognizes. Instructions sometimes requires anywhere from 0 to 2 arguments to complete the command. A list of instructions can be found below in the Instruction Set section.
Register This consists of any of R0, R1, R2, and R3. The register can only be used as an argument.
Immediate

Immediates are started with a dollar sign "$" and are followed immediately by a number. An immediate is a constant numeric value in either hexadecimal (prepended with a 0x) or decimal form. Given values exceeding 16 bits in size, the emulator will take the 16 least significant bits.

Decimal immediate:

$1234

and hexadecimal immediate:

$0xBEAF
Address

Addresses are represented as numbers without any decorations. Addresses are available in both hexadecimal and decimal format. Addresses are bound at address 0 at the minimum and 511 as the maximum.

Decimal address:

123

and hexadecimal address:

0x00AF
Label

Labels are started with a period sign ".", are followed immediately by a name, and must be placed at the beginning of a line. Labels act as a way of naming numbers. They can be used in place of any number (both addresses and immediates).

.SUM_FUNCTION

All labels names must be unique.
Comment Comments are started with a semicolon ";" and continue until the of the line. Comments are used to annotate and describe the code at hand.

Syntax Format

Line Delimited

This assembly language is line delimited meaning that each new line indicates that there's a new instruction. In other words, putting multiple instructions on a single line will fail to assemble. It is also generally considered to be bad style to have multiple instructions per line.

Comma Delimited Arguments

For instructions that have greater than one argument, a comma is used to separate the arguments.

Here is a generic template for all instructions:

.optional_label INSTRUCTION arg0, arg1 ; Comment which is optional
Register Uses
    POP R0
    SET R0, R1
Immediate Uses
    SET $1234  , R0
    SET $0xBEAF, R1
Address Uses
    MOV R0    , 500
    MOV $0x123, 0x100
Label Uses

To declare a label, it is also acceptable to place the label in the line above the target code. 

    .LABEL       ; Label being set
    SET 0xA, R0  ; Ensuing instruction(s)
    ...

To declare a memory location with a label, place the value after the label. 

    .LABEL 200       ; .LABEL is now associated with 200

Labels can then be used memory would, for instance to branch:

    .MUL_FUNCTION
    MUL R0, R1
    RTN
    ... ; A couple lines later...
    BRN .MUL_FUNCTION

and can be assigned to an address and used as memory as follows:

    .ADDRESS_NAME 100
    MOV R0, .ADDRESS_NAME

Labels can be used even if they are declared after being used in their instruction for instance:

    BRN .MUL_FUNCTION
    ... ; A couple lines later...
    .MUL_FUNCTION
    MUL R0, R1
    RTN

Labels can also be used in immediates, for instance:

    .VALUE 100
    SET $.VALUE, R0 ; Puts 100 into R0

PC and SP

The PC and SP fields indicates what happens to those registers upon completion of the instruction.

ZCNO

The ZCNO field changes are indicated by the flags that get changed, hence potentially any combination of Z, C, N, and O.

Condition Status Symbol Description
- Does not influence this status bit.
? Conditionally influences the status bit.
0 Sets the given status bit to 0.
1 Sets the given status bit to 1.

Example:

Given an instruction with the following ZCNO code 0?-1, means that the instruction sets the zero bit to zero, conditionally sets the carry bit, doesn't change the negative bit, and always asserts the overflow bit.

Byte Code

There is also a field indicating the byte code(s) of the instruction. A single instruction will have multiple byte codes when it accepts more than a single argument type for any of its arguments. The number of byte codes that such an instruction has is detailed in the table below:

# of Argument(s) # of Byte Code(s)
0 1
1 # of argument types arg0 accepts
2 (# of argument types arg0 accepts) X (# of argument types arg1 accepts)

Example:

The SET instructions' arg0 accepts three argument types: I, R, and M. SET's arg1 accepts only a single type, R. Hence, SET has 3 byte codes, one for the I paired with R, R paired with R, and M paired with M.

Mapping Byte Code

The mapping of the instruction by argument type to its respective byte code is shown below: 

Given a targeted instruction, "ins".

1) Take the index of your desired arg0 from the available options for "ins". Let this index be termed "a".

2) Take the index of your desired arg1 from the available options for "ins". Let this index be termed "b".

3) Take the number of possible argument options for arg1 and term this "b_length".

4) The index of the target opcode is: a * b_length + b

Example:

Perform CMP on a register as arg0 and memory as arg1.

              a = 2
              b = 1
       b_length = 3
index of opcode = 2 * 3 + 1
                = 7
         opcode = 25
